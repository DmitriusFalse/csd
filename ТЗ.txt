 Техническое Задание (ТЗ): "Civitai Smart Downloader"
1. Цель проекта
Создать комплексное решение (Браузерное расширение + Локальное приложение) для автоматизированного скачивания моделей (Lora, Checkpoints, Embeddings) с сайта Civitai (и его зеркал) напрямую в заданные директории с последующей интеграцией с Lora Manager.
1.1. Сайты откуда качать
 civitai.com, и civitai.red
Это два клона сайта, работают идентично.

2. Архитектура системы
Система состоит из двух модулей, общающихся по локальной сети:
Модуль А (Browser Extension): Расширение для Chromium-браузеров (Manifest V3). Отвечает за UI, извлечение ID модели со страницы и отправку команд.
Модуль Б (Local Bridge): Локальный микросервис. Отвечает за загрузку файлов, работу с файловой системой ОС и взаимодействие с Lora Manager.
3. Требования к Модулю А (Браузерное расширение)
3.1. Пользовательский интерфейс (UI)
Инъекция на страницу: При заходе на страницу модели расширение добавляет кнопку [⬇ Скачать в Lora Manager] рядом со стандартными кнопками сайта.
Popup-окно (Настройки):
Поле для ввода API-ключа Civitai (необходим, так как сайт защищает ссылки от парсинга и требует авторизацию для скачивания).
Поле для указания порта локального сервера (по умолчанию localhost:8765).
Выбор типа модели (Lora, Checkpoint, VAE), если расширение не может определить его автоматически.
Уведомления: Всплывающие сообщения (Toast) о статусе («Задача отправлена на ПК», «Ошибка связи с сервером»).
3.2. Логика и Парсинг
Content Script: Анализирует страницу, чтобы извлечь:
modelVersionId (ID конкретной версии модели для API).
modelName (Название файла).
modelType (Тип модели).
Background Worker: Формирует JSON-запрос и отправляет его на локальный сервер.
4. Требования к Модулю Б (Локальный сервер)
4.1. API для приема задач
Эндпоинт POST /download, принимающий JSON:
{
  "modelVersionId": 290640,
  "fileId": 228616,
  "modelType": "Checkpoint",
  "baseModel": "SDXL",
  "modelName": "Pony Diffusion V6 XL",
  "fileName": "ponyDiffusionV6XL_v6.safetensors",
  "fileSize": "6.46 GB",
  "apiKey": "civitai-api-key-here"
}

4.2. Загрузчик файлов (Downloader)
Формирование прямой ссылки: https://civitai.com/api/download/models/{versionId}?token={apiKey}.
Скачивание чанками (кусками), чтобы не перегружать оперативную память при загрузке моделей по 5-10 ГБ.
Поддержка Resume (возобновление скачивания при обрыве связи).
Опционально: Использование aria2c для многопоточного скачивания (ускоряет загрузку в 3-5 раз).
4.3. Маршрутизация по папкам
Чтение конфигурационного файла (.json или .yaml), где прописаны пути:
 "См. раздел 9.3"
4.4. Интеграция с Lora Manager
После успешного сохранения файла сервер должен "пингануть" Lora Manager, чтобы тот обновил базу.
Варианты: Отправка HTTP-запроса на локальный API менеджера или запуск пользовательского .bat / .sh скрипта (Webhook) после завершения загрузки.
5. Обработка ошибок и Безопасность
Безопасность: Локальный сервер должен слушать только 127.0.0.1 (localhost), чтобы исключить доступ к нему из внешней сети.
Ошибки API:
401 Unauthorized: Неверный API-ключ (уведомить пользователя).
404 Not Found: Модель удалена.
429 Too Many Requests: Cloudflare/Civitai ограничило частоту запросов (поставить задачу в очередь с повтором через N секунд).
Автостарт: Локальный сервер должен уметь сворачиваться в системный трей и запускаться вместе с Windows/macOS/Linux.

Основной эндпоинт получения данных модели:

Возвращаемые данные (предположительно):
Полное название модели

/api/trpc/user.getCreator - информация о создателе модели
/api/trpc/track.addView - отслеживание просмотров (не нужен для скачивания)
8.5. Авторизация и безопасность
Обнаруженные механизмы:
Куки-авторизация: Все API-запросы отправляются с credentials: "include", что означает автоматическую передачу сессионных кук
Параметр authed: В каждом tRPC-запросе передается "authed":true для проверки авторизации
API-ключ: Используется в query-параметре ?token={apiKey} для программных запросов
Заголовки запросов:
sec-ch-ua: "Chromium";v="142"
sec-fetch-site: same-origin
credentials: include

10.3. Безопасное хранение API-ключа
10.3.1. Архитектура безопасности
Принцип: API-ключ хранится только в локальном Go-приложении, расширение его не видит.
[Браузерное расширение]
        ↓ (запрос без ключа)
[POST localhost:8765/download]
        ↓
[Go Local Bridge]
        ↓ (добавляет ключ из зашифрованного хранилища)
[GET civitai.com/api/download/models/{id}?token={key}]
10.3.2. Шифрование API-ключа
Метод: AES-256-GCM с ключом, производным от машинного ID.
Процесс:
Первый запуск приложения:
→ Пользователь вводит API-ключ через Tray-меню
→ Генерируется машинный ID (хеш от MAC-адреса + hostname)
→ Ключ шифруется AES-256-GCM
→ Сохраняется в config.yaml как зашифрованная строка
Последующие запуски:
→ Приложение вычисляет машинный ID
→ Дешифрует API-ключ из config.yaml
→ Использует для запросов к Civitai
Пример записи в config.yaml:

api_key_encrypted: "aes256:aGVsbG8gd29ybGQ=..."  # Зашифрованный ключ
machine_id_hash: "sha256:abc123..."
10.3.3. Смена API-ключа
Процесс через Tray-меню:
⚙ Настройки → Сменить API-ключ
→ Открывается модальное окно (или браузер на localhost)
→ Пользователь вводит новый ключ
→ Старый ключ перезаписывается новым (шифрованным)
→ Уведомление: "🔑 API-ключ обновлен"
10.3.4. Защита от XSS
Проблема: Если расширение хранит ключ в chrome.storage.local, XSS-атака на сайте Civitai может его украсть.
Решение: Расширение никогда не хранит и не видит API-ключ. Оно просто отправляет запрос на localhost:8765, а ключ подставляется на сервере.
Дополнительная защита:
Go-сервер принимает запросы только с 127.0.0.1
Добавление секретного заголовка X-Bridge-Secret (генерируется при установке расширения)
Rate limiting на локальном сервере (максимум 10 запросов в минуту от одного источника)

10.5. Механизмы интеграции с Lora Manager
10.5.1. Варианты интеграции
Конфигурация:
lora_manager_integration:
  enabled: true
  
  # Вариант 1: HTTP Webhook
  webhook_url: "http://localhost:7860/api/models/refresh"
  webhook_method: "POST"
  webhook_headers:
    Authorization: "Bearer your-token"
    Content-Type: "application/json"
  webhook_body_template: |
    {
      "action": "model_added",
      "model_path": "{{.ModelPath}}",
      "model_type": "{{.ModelType}}",
      "metadata": {{.MetadataJSON}}
    }
  
  # Вариант 2: Запуск скрипта
  on_complete_script: "C:\\Scripts\\refresh_lora_manager.bat"
  script_args:
    - "{{.ModelPath}}"
    - "{{.ModelType}}"
  
  # Вариант 3: Мониторинг файловой системы (Lora Manager сам увидит)
  file_watcher_enabled: false  # Lora Manager использует fsnotify
10.5.2. Сценарии интеграции
Сценарий 1: Webhook (рекомендуется)
1. Загрузка завершена
2. Go-сервер отправляет POST-запрос на webhook_url
3. Lora Manager получает уведомление
4. Lora Manager сканирует папку и добавляет модель в базу
5. Модель появляется в интерфейсе Lora Manager
Сценарий 2: Скрипт

1. Загрузка завершена
2. Go-сервер запускает on_complete_script с аргументами
3. Скрипт выполняет необходимые действия (например, curl к API Lora Manager)
4. Lora Manager обновляет базу
Сценарий 3: File Watcher

1. Загрузка завершена
2. Файл сохраняется в папку
3. Lora Manager (использующий fsnotify) автоматически обнаруживает новый файл
4. Lora Manager сканирует и добавляет модель
10.5.3. Обработка ошибок интеграции

Если webhook недоступупен:
→ Попытка 1: Отправить webhook
→ Ошибка: Connection refused
→ Логирование: "⚠ Webhook недоступен, повтор через 30 сек"
→ Попытка 2 (через 30 сек): Отправить webhook
→ Успех → Уведомление: "✅ Lora Manager обновлен"
Максимум попыток: 3, затем уведомление пользователя о необходимости ручной проверки.

10.7. Управление версиями и обновлениями
10.7.1. Версионирование приложения
Схема версионирования: SemVer (MAJOR.MINOR.PATCH)
Пример: 1.2.3
1 — мажорная версия (несовместимые изменения API)
2 — минорная версия (новая функциональность)
3 — патч (исправления багов)
Хранение версии:
// В коде
const AppVersion = "1.2.3"

// В config.yaml
app_version: "1.2.3"
config_version: "1.0"  # Версия схемы конфига
10.7.2. Автоматические обновления
Механизм: GitHub Releases + github.com/rhysd/go-github-selfupdate
Процесс:
1. При запуске приложения (или раз в 24 часа)
2. Проверка последней версии на GitHub Releases
3. Если новая версия доступна:
   → Уведомление: "🔄 Доступна новая версия 1.3.0 (текущая: 1.2.3)"
   → Кнопки: [Обновить сейчас] [Позже] [Не спрашивать]
4. При нажатии "Обновить сейчас":
   → Скачивание нового бинарника
   → Замена старого файла
   → Перезапуск приложения
Конфигурация:

updates:
  auto_check: true           # Автоматически проверять обновления
  check_interval_hours: 24   # Интервал проверки
  auto_download: false       # Автоматически скачивать (требует подтверждения)
  github_repo: "username/civitai-smart-downloader"
10.7.3. Миграция конфигурации
Проблема: При обновлении приложения структура конфига может измениться.

Пример:
# Старый конфиг (config_version: 1.0)
paths:
  lora: "D:\\Models\\Lora"

# Новый конфиг (config_version: 2.0)
paths:
  root: "D:\\Models"

if configVersion == "1.0" {

}

10.8. Дополнительные требования к надежности
10.8.1. Graceful Shutdown
Сценарий: Пользователь закрывает приложение во время активной загрузки.
Поведение:
1. Получен сигнал SIGTERM/SIGINT
2. Пауза всех активных загрузок (сохранение прогресса)
3. Ожидание завершения текущих операций (максимум 10 сек)
4. Сохранение состояния очереди в queue.json
5. Корректное завершение работы
6. Уведомление: "⏸ Загрузки приостановлены. Возобновятся при следующем запуске"
10.8.2. Восстановление после сбоя
Сценарий: Приложение упало во время загрузки.
Поведение при следующем запуске:
1. Чтение queue.json
2. Обнаружение незавершенных загрузок
3. Уведомление: "🔄 Найдено 2 незавершенные загрузки. Возобновить?"
4. При подтверждении:
   → Проверка наличия частично скачанных файлов
   → Возобновление с последней точки (используя Range-заголовки)

// Размер чанка: 1 MB
const chunkSize = 1024 * 1024

// Скачивание чанками с записью в буфер
buffer := make([]byte, chunkSize)
for {
    n, err := response.Body.Read(buffer)
    if n > 0 {
        file.Write(buffer[:n])
        downloadedBytes += int64(n)
        updateProgress(downloadedBytes, totalBytes)
    }
    if err == io.EOF {
        break
    }
}

10.9.3. Кэширование метаданных
Проблема: Повторные запросы к API для уже скачанных моделей.
Решение: Локальный кэш в SQLite или BoltDB.
cache:
  enabled: true
  storage: "bolt"              # bolt или sqlite
  path: "cache/metadata.db"
  ttl_hours: 168               # Время жизни кэша (7 дней)
Использование:

// Перед запросом к API проверяем кэш
if cached := cache.Get(modelVersionId); cached != nil {
    return cached, nil
}

// Если нет в кэше — делаем запрос к API
metadata := fetchFromAPI(modelVersionId)
cache.Set(modelVersionId, metadata)
return metadata, nil
